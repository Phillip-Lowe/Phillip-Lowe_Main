## Problem

SAOS dashboard system was burning compute by repeatedly spawning SOL sessions with vague "client_request" tasks. User noticed this had happened before (June 24 orchestrator loop incident) and wanted it stopped + fully documented for re-enable later.

---

## Investigation Results

### Sources of Wasted Compute Found

| # | Source | Impact | Status |
|---|--------|--------|--------|
| 1 | **Orchestrator Daemon** (PID 1076) | Spawning SOL every ~10s for 14 pending tasks | **Killed + LaunchAgent unloaded** |
| 2 | **n8n Email Dispatcher** (`eylye0Me5zyoXMc2`) | 1,440 failed executions/day (wrong API port) | **Disabled in DB** |

### Root Cause
- Orchestrator daemon built June 17, credential paths broke after June 22 workspace reorg
- 14 test `client_request` tasks (IDs 325-336) left in PENDING state from June 23
- Daemon restarted on boot because LaunchAgent `net.systack.orchestrator` was never unloaded after June 24 fix
- n8n Email Dispatcher pointed to `localhost:8768` instead of `localhost:8765` (wrong API port)

## Prevention

- Before re-enabling daemon: verify credential paths in `scripts/provision_vps.py`
- Before re-enabling daemon: clear task_queue of old test tasks
- Before re-enabling email dispatcher: fix API port from 8768 to 8765

- Add pre-flight check: daemon verifies credentials before starting loop
