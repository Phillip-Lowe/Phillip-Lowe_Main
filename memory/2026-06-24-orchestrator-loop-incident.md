# Incident Report: Orchestrator Daemon Runaway Loop

**Date:** 2026-06-24
**Time:** ~03:07 CDT
**Severity:** HIGH — Burned compute for ~18 hours
**Status:** RESOLVED

---

## What Happened

The SAOS Orchestrator Daemon (`orchestrator-daemon.py`) was running continuously since June 17, processing tasks from a Postgres task_queue. It spawned sessions that kept failing because **credential paths were wrong** after a workspace reorganization.

### Timeline

- **June 17:** Daemon built and deployed via LaunchAgent (`net.systack.orchestrator.plist`)
- **June 22:** Workspace reorganization moved credentials from `credentials/Green/` to `Sol-Knowledge/credentials/Green/`
- **June 22:** Vultr VPS test succeeded with corrected paths, but daemon paths were NOT updated
- **June 22–June 24:** Daemon kept trying tasks, failing on credential load, retrying, burning compute
- **June 24 03:07:** User noticed BlueBubbles webchat spam (Tasks #205–#207)
- **June 24 03:22:** Daemon stopped, LaunchAgent unloaded
- **June 24 03:35:** Credential paths fixed in ALL scripts

### Damage

| Metric | Value |
|--------|-------|
| Failed tasks (SOL) | 230 |
| Completed tasks (SOL) | 152 |
| Pending tasks | 101 |
| Dead tasks | 61 |
| Task files generated | 368 |
| Compute burned | ~18 hours of local MacBook Air |

### Root Cause

1. **Workspace reorganization broke paths** — credentials moved but scripts weren't updated
2. **No cleanup after reorganization** — no check that daemon still worked
3. **LaunchAgent respawn** — daemon kept restarting on crash, masking the failure
4. **No path validation** — scripts didn't fail gracefully when credentials were missing

### Immediate
1. Killed orchestrator daemon process
2. Unloaded LaunchAgent (`net.systack.orchestrator.plist`)
3. Verified no more runaway sessions

## Prevention

1. **Never deploy daemon with hardcoded paths** — use environment variables or config file
2. **After ANY workspace reorganization** — run credential path validation
3. **Add pre-flight check** — daemon should verify credentials before starting loop
4. **Monitor task failure rate** — if >50% tasks fail, alert and stop
